Overview

Primary Assignment Objective

The primary objective of this assignment is to explore some operator overloading, e.g., +, -, *, /. +=, -=, *=, /=, ==, and <=>. Later on this page is a provided C++ program called a2-provided.cpp. In that file is a complete C++ program except for some:

#error "Wirte code here."

preprocessor directives. You will replace those preprocessor directives with the correct code. The rest of the program is to remain unmodified. (The rest of the program is 100% correct: you only need to write code where indicated.)

Secondary Assignment Objective

The secondary objective of this assignment is, after finishing the assignment and handing it in, to continue to explore the provided code for C++ language features used, looking up things in the textbooks, and/or cppreference.com.

Program Synopsis

Notice in the provided program that main(), found at the bottom of the file, calls two functions:

  • simple_rel_demo(), and,
  • simple_rpn_demo().

The simple_rel_demo() function reads in an input file and stores all unsigned integer values read in into a std::vector<mod5> object. After such it processes the object outputting the results of applying ==, <, and > to every pair of values in the vector producing output as it does those comparisons.

ASIDE: This is will allow you to see whether or not the == and <=> definitions you wrote are correct. Correct output can be see in the Sample Program Runs section later on this page. This will also allow you to explore how <=> generates functions. For example, is the output correct if you (after finishing and handing in the assignment first!) define ==, <, and <=> more naively/lazily? etc.

The simple_rpn_demo() function reads in an input file pushing all unsigned integers read in (as mod5 objects) onto a stack and if valid operators (i.e., +, -, *, and /) are encountered, it applies those operators to the stack. Exceptions that occur during stack processing cause an error message to be output to std::cerr and processing continues. The processing continues until the input file is consumed or the stream is no longer good. After such, the stack's content is dumped to std::cout.

That said, this program does something special: instead of using unsigned integers, it processes "mod5" class instances. Each mod5 class holds an unsigned integer and treats that integer in a special way: for the purposes of externally accessing its unsigned value, it outputs the reminder of the unsigned value divided by 5, i.e., it represents modulus arithmetic where the modulus is 5.

The +, +=, -, -=, *, and *= operations are well-defined under modulus arithmetic, thus, these operations simply add, subtract, and multiply the operands --as there is no need to compute % 5. (ASIDE: unsigned integers are permitted to overflow and when that occurs it is a modulo operation, so there is no need to worry about overflow / wrap-around.)

The / and /= operations are not well-defined under modulus arithmetic --but we want a useful rpn calculator (mod 5) --so division is defined to be if mod5::mod_value() returns 0 for the denominator, an exception must be thrown (since that would result in a divide by zero), but, when computing the division operation, one mustn't use mod_value(), instead one must use the actual value of mod5::i in the numerator. (The denominator must be the mod5::mod_value() quantiy.)

Writing operator+=, operator-=, operator*=

Each of these operator definitions are exactly two lines of code, i.e., +=, -=, and *= this->i with that of rhs.i, and, be sure to return *this.

NOTE: Assignment operators assign the left-hand side to whatever is computed using the right-hand side. This implies that the assignment operator's left-hand side (i.e., "*this" in this case since these operators are being defined as member functions of the mod5 class) must return an lvalue (i.e., so it can be assigned). Since object-oriented "this" existed in C++ before references were brought in to the language, "this" is instead a constant pointer. This is why *this needs to be used with return *this;, and, this-> is used to access members, etc. --because this, itself, is a pointer. :-)

Writing operator/=

ASIDE: This is the most complicated function to write in this assignment.

If the right-hand side of /= mod_value() is 0 (zero), then dividing will result in a divide by zero --and this needs to be avoided. It will be avoided by throwing an exception. Before throwing an exception however a string holding the reason for the exception will be built using an std::ostringstream object. (Recall in C there was sprintf() which operated on strings. In C++, std::ostringstream does the same except one does not need to worry about dynamic memory allocation, etc.) The exception that will be thrown is a std::domain_error. To do this, ensure the following is in the appropriate clause of your if statement:

std::ostringstream buf;
buf << "mod5 divide by zero; operands(" << i << ' ,' << rhs.i << ')';
throw std::domain_error(buf.str());

Notice that using std::ostringstream is the same as using std::cout, std::cerr, and std::clog. (The same applies for std::istringstream and std::cin. There is also a std::stringstream which can do both input and output.)

After the if statement, perform the division using the value of this->i in the numerator and rhs.mod_value() in the denominator and ensure the result is assigned to this->i. Then properly return from the function.

Writing operator==

A correct implementation of this function is a single line of code.

First, realize this function is inside the mod5 class --which is, obviously, currently being written. Already written are the types mod5 is using, e.g., unsigned for its data member i. Since unsigned is already written (it is, in fact, provided by the C++ language) and has a definition for ==, it is easiest and best to leverage that definition in writing the code for this function! Don't reinvent the wheel!!! :-)

The single line of code to write for this function is to compare the left-hand side argument's mod_value() with the right-hand side argument's mod_value() using operator ==. (ASIDE: Recall that relational operators in C++ return bool values (true or false). Use of relational operators in C++ do NOT need to be in an if, while, for, do-while, etc. statement to use them. Do NOT write an if..else statement or use the ternary operator in this function! You only need a single line of code: operator==() returns a bool, so use it. :-)

ASIDE: Another reason to NOT use an if..else statement is that an if..else statement is an explicit "branch" instruction. Branch instructions can significantly and negatively impact modern CPU instruction pipelining performance. By only writing the test and returning its bool result, allows the compiler/programmer to better determine how to make use of the boolean value. For example, for speed reasons, the programmer, yes, might prefer storing 1,000,000 bool values in a std::vector so that CPU SIMD vector instructions could be used to do the relational operator use (as those only result in bool values and modern CPUs can typically do such with SIMD instructions if the bool value itself is not used in a branch within the SIMD instruction) very quickly in parallel. (After such, presumably the programmer would do something with those bool values.) Had an if statement been used, SIMD vector instruction use would not have been generated by the compiler. In a number of programs, yes even with using that much RAM for bool values, such can easily result in faster code.

Writing operator<=>

A correct implementation of this function is a single line of code. Notice:

  • there is no = defaut, and,
  • the return is auto.

No = default means you are writing a definition of this function, and, the auto return tells the compiler to determine what the appropriate return type (i.e., ordering) is based on the code you write in this function.

First/Again, realize this function is inside the mod5 class --which is, obviously, currently being written. Already written are the types mod5 is using, e.g., unsigned for its data member i. Since unsigned is already written (it is, in fact, provided by the C++ language) and has a definition for <=>, it is easiest and best to leverage that definition in writing the code for this function! Don't reinvent the wheel!!! :-)

Now, write the code for this function. The code is a return statement returning the three-way-comparison of the left-hand side's mod_value() and the right-hand side's mod_value(). (Clearly such will return some kind of ordering and that type will be used by this function as the return type is auto. unsigned integers have std::strong_ordering.)

Writing operator+, operator-. operator*, and operator/

Each of these operators is exactly three lines of code:

  • The first line creates a local copy of the left-hand side object.
  • The second line invokes the corresponding +=, -=, *=, and /= operator on the local copy with the right-hand side object.
  • The last line returns the local copy.

ASIDE: This is why the +=, -=, *=, and /= objects were written first. The +, -, *, and / code only needs to be written once and other operators can simply call those definitions elsewhere. This makes it much easier to write and test code, and, it is easier to fix bugs since, fundamentally, the +, -, *, and / operations are defined on in one location. :-)

Writing operator<<

This function is exactly two lines of code:

  • Use operator << to output the mod_value() of the mod5 object passed in to the std::ostream object.
  • Return the std::ostream object.

ASIDE: As with all IOStream operator<< definitions, the stream is always passed by reference as the left-hand side parameter and it is always returned by reference. The right-hand side parameter is always passed in as a constant reference (const&).

Writing operator>>

Reading in a mod5 value is to be done in a way that if the mod5 object is not properly read in, the stream is marked as "fail". This induces processing on the stream to stop due to a failure of reading in the data. This is a good thing since if the data was not properly read in, one shouldn't be using processing such --the datum is invalid!

First declare a local unsigned integer, called temp. (This variable will be used to read in the next unsigned integer value.)

If the expression is >> temp is true, then assign the right-hand-side variable to mod5{temp}, otherwise, call is.setstate(std::ios_base::failbit).

Finally, return the stream variable from the function.

ASIDE: The code in this function does NOT assign this function's right-hand-side variable unless it is properly read in. Code written this way allows the caller to be able to know that if the datum is not properly read in, the variable the datum is to be stored in is left as-is. (Clearly the stream is not --but that is to be expected with a stream.) This makes it easier to deal with issues in one's program, e.g., the caller does not need to make a potentially expensive copy of a variable before calling the function. (Such would need to be done if the variable was modified even if it wasn't properly read in.)
