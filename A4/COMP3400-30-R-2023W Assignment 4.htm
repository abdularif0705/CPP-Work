Overview

Playing cards are very flexible. With a set of 52 standard playing cards + 2 jokers (red and white), a plethora of card games can be played. Each of these games use different cards from the desk (if not all of them), sorts/ranks/values them differently, etc. This makes such useful to explore some C++ language constructs, Standard Library algorithms, as well as sorting/ranking (especially since each card game, if not also within that game, the sorting and ranking will be different from many other (if not all) card games.

In this assignment you will:

  • write code to read and write card face values to/from a stream, and,
  • write a basic playing_card class.

While straight-forward, there are some restrictions on how this is to be done:

  • most I/O must be done using unformatted stream operations (see below),
  • all input (read) operations must be done in a way that no card_face object is created if a vaiid card_face object wasn't read in,
  • card_suit and card_face types are provided and are implemented as enum class types,
    • (One may or may not wish to do such with card_face --in this assignment they are enum class types.)
  • the playing_card type must have a default, lexicographical, strong ordering,
    • (This program is not implementing a card game, rather, it is lower-level and needs to provides enough to sort cards.)

Details concerning such is below. Additionally, some of the provided code should be examined to glean/learn/consider some implementation ideas. For example, card_suit is implemented as an enum class and read_card_suit(), operator<<(), and operator>>() are implemented for such in the provided code. The card_face enum class is more elaborate but you are required to write code for read_card_face(), operator<<(), and operator>>() for card_face --much of it is very similar to the code for card_suit.

Finally, main(), in this assignment, is provided to you. (The next assignment will explore more using the Standard Library's algorithms, etc.) That said, white such is provided to you, while developing your code you might not want to immediately use that main program. One reason: you'll need to have most things implemented correctly before the code in main() will run properly. Instead, start off with an empty main() and write code to read/write card_face's, etc. to test your code. Only once such is good, then consider using the provided main().

Premble

Before starting to write code for this function:

  • look at read_card_suit(std::istream&) in at-provided.cpp
  • Section 15.4 Constants for the State of Streams in TCPPSL.
  • Section 16.1.4 Character Traits in TCPPSL.
    • NOTE: Every IOStream class has a traits_type (type alias) member which is of type std::char_traits for the character type used with the stream. Additionally, know every IOStream class is actually a class template, e.g., std::basic_istream, and one of the template parameters is the character type used in the stream. So while you've been using std::istream, std::ostream, etc. know that those are type aliases to std::basic_istream<char> and std::basic_ostream<char> respectively.

The IOStreams classes are indeed classes with member functions, etc. Amongst other functions, the std::basic_istream class defines .get() and .unget(). Analogously std::basic_ostream defines .put() (there is no "unput()"). These calls correspond to C's getc(), ungetc(), and putc() and are part of some of the unformatted I/O operations that can be performed. In C++, the unformatted operations are these:

  • for std::basic_istream: https://en.cppreference.com/w/cpp/io/basic_istream#Unformatted_input
  • for std::basic_ostream: https://en.cppreference.com/w/cpp/io/basic_ostream#Unformatted_output

whereas the operations done using operator>>() and operator<<() are considered formatted operations since they typically do things such as skip whitespace, can format data in special ways (e.g., precision, width settings), etc. Other unformatted calls include .read() and .write() which correspond to C's fread() and fwrite() functions.

This assignment intentionally uses the unformatted operations to draw attention to such calls --which are low-level calls or used when reading/writing binary data is needed, i.e., the data read/written is to be done as is --not formatted in any way.

The std::char_traits class defines how to perform a variety of essential operations with an arbitrary character type. Not all characters are of type char: there is char, unsigned char, signed char, wchar_t, char8_t, char16_t, and char32_t. (Unfortunately, wchar_t should have been what char32_t is as wchar_t represents the largest character type supported, e.g., UCS-32 (Unicode 32-bit), but unfortunately Microsoft's compiler used UCS-16 (Unicode 16-bit) for wchar_t which cannot represent all Unicode characters. Since much of the planet uses Windows and a lot of code has been written this way, this essentially has forced the ISO C++ standard (and other standards bodies) to create new definitions (i.e.,in ISO C++ char8_t, char16_t, and char32_t) to support Unicode characcters. Fortunately, C++ in its early days recognized that there would be more character types than simply char and virtually all relevant C++ Standard Library definitions have template parameters for the character type, e.g., std::basic_istream, std::basic_ostream, std::basic_string, std::basic_regex, etc. If any of these (and other names) are similar but new to you, it is because there are type aliases defined as synonyms to such types using type char, e.g., std::istream, std::ostream, std::string, std::regex, etc.

In this assignment we are not interested in writing code that works for any character type, instead we are interested in writing code using IOStreams' unformatted member functions. In C, EOF is used to identify the end-of-file or an error on the stream (feof() is used to actually confirm the end of a file in C). Also many learn that EOF, which is a macro in C, is commonly set to a value of -1 --although such can vary from compiler to compiler (always use the macro). While this seems unimportant, it is extremely important. Consider:

  • -1 is a literal of type int
  • 'a', etc. are literals of type char
  • char values can be implicitly converted to int

Suppose an implementation represents a char as an 8-bit unsigned value, i.e., 0 to 255, then if a char value is implicitly converted to an int, then those integer values would be in the range from 0 to 255. If EOF == -1, then EOF is a distinct value outside of the range of values of 0 to 255, thus, the EOF value will never conflict with a character value --allowing errors to be identified via EOF (and any other value not equal to a character value). C++'s IOStreams does something similar --but generically (i.e., with templates):

  • std::char_traits<typename Char> for a given Char type (in a low-level manner):
    • defines what it means to assign a Char value to another Char value
    • defines what it means to perform an equality comparison between two Char values
    • defines what it means to perform a less than comparison between two Char values
    • defines what it means to move one Char sequence into another
    • defines what it means to copy one Char sequence into another
    • defines what it means to compare two Char sequences
    • defines what it means to determine the length of a Char sequence
    • defines what it means to convert "int_type" to an equivalent Char type
    • defines what it means to convert a Char type to an equivalent "int_type"
    • defines what it means to compare two "int_type" values
    • defines what eof() is (i.e., such returns an end-of-file int_type value for Char type)
    • defines what not_eof() means

i.e.,

  •   is requivalent to EOF in C
  • The type of std::char_traits<Char>::eof() must be std::char_traits<Char>::int_type, i.e., int, e.g., Char is the type char.

So while writing std::char_traits<char>::eof() is more verbose than writing EOF, such due to the use of templates can handle any character type std::char_traits<T> is partially specialized for, e.g., char, wchar_t, char8_t, char16_t, and char32_t in ISO C++. (C's EOF cannot be used for any type where its value would overlap the range of values representable by the character type. If one uses std::char_traits in C++, one is "future-proof" regardless of the character type being used. The latter implies the code will not need to be rewritten, debugged, etc. with a new character type.)

In this assignment you will need, just as is done in a4-provided.cpp, determine if the character read using .get() (e.g., see https://en.cppreference.com/w/cpp/io/basic_istream/get) was a valid character or an error/EOF value. Notice that .get() returns int_type --which corresponds to std::char_traits<char>::int_type, i.e., see https://en.cppreference.com/w/cpp/string/char_traits. (Effectively, when the character type is char, this is the same as C.) The .get() member returns std::char_traits' eof() value when there is an error/end-of-file, so this means to check if the value returned from .get() is an error/end-of-file or not one needs to write:

std::istream::traits_type::eq_int_type(ch,std::istream::traits_type::eof())

using std::char_traits' definitions:

  • ch is a value returned by .get() (i.e., some int_type)
  • eof() is the error value / end-of-file value for that char_type (and its type is int_type)
  • eq_int_type() compares two int_type values
  • traits_type is a type alias to the std::char_traits for that stream type.

So while, yes, writing:

if (!std::istream::traits_type::eq_int_type(ch,std::istream::traits_type::eof()))

is much longer than writing in C:

if (ch != EOF)

the C++ version will work regardless of the character type for the stream. In this program we are using the std::istream and std::ostream which correspond to std::basic_istream<char,std::char_traits<char>> and std::basic_ostream<char,std::char_traits<char>> respectively. Had this assignment been using the latter with a template parameter, CharT (defined elsewhere), instead of hard-coding the type char, then the code would be generically written for any character type, e.g.,

if (!std::basic_istream<CharT>::traits_type::eq_int_type(ch,std::basic_istream<CharT>::traits_type::eof()))

(To be clear such is not being done in this assignment. That said, library code is often written like this to ensure such code works with any character type.)

The read_card_suit() has two more function calls:

  • .unget()
  • .setstate()

The unget() call returns the last read character back to the stream (buffer). ISO C++ only guarantees this will work for one character only. (ASIDE: If one wants to support more than one "unget" character, then one needs to implement his/her own implementation of std::basic_streambuf. How to do this is beyond the scope of this course although TCPPSL discusses streambufs in some detail should you be interested in such.)

The setstate(val) call allows one to change the stream state to some specified val value. These values will be one of goodbit, failbit, badbit, or eofbit. Typically setstate() is used to set failbit or badbit in user code. (Either will work. Generally programmers will use failbit for recoverable errors and badbit for non-recoverable errors although this is only a convention.)

Finally, there are two more useful functions:

  • .peek(): see https://en.cppreference.com/w/cpp/io/basic_istream/peek
  • .ignore(): see https://en.cppreference.com/w/cpp/io/basic_istream/ignore

not used in read_card_suit() that you will use when you write read_card_face().

The peek() function behaves like .get() except it does not remove it from the input buffer. (All bufferred file I/O uses an intermediate buffer. The .peek() function allows one to read the next character which is left unconsumed in the buffer. For those who took COMP2140 (the Languages, Grammars, and Translators course), .peek() is a look-ahead-one-character function. The .peek() function is useful because one can examine the next character without consuming it. For example, suppose the next character may or may not be a digit. One can determine if it is a digit or not without removing it from the read buffer by calling .peek().

The ignore() function allows one to ignore (by default 1) character(s) on the input stream. For example, suppose one used peek() and wanted to consume the character. One could call .get() and ignore the returned result, or, one could call .ignore() which effectively does that (i.e., calls .get() and ignores the result).

Writing read_card_face()

WIth such in mind, write read_card_face(), which will be similar to read_card_suit(), except with some more logic required:

  • At the top of the function write std::optional<card_face> retval;
    • i.e., if the function successfully reads in a card_face, then retval will be set, otherwise it will not be. This is what std::optional<T> is designed to represent: having a value or not having a value.
  • Call the input stream's .get() member function. This variable will be referred to as 'ch' below.
    • Tip: Declare the variable to hold the result as auto. This saves you from having to write std::istream::traits_type::int_type instead. :-)
  • Check if ch is the end-of-file value or not. If it is not, then:
    • Using a switch statement (this is easier) with case statements corresponding to 'A', '2', ... '9', 'J', 'C', 'Q', 'K', 'R', 'W', and '1' set retval as follows:
      • 'R' means card_face::red_joker
      • 'W' means card_face::white_joker
      • 'C' means card_face::knight
      • 'A' means card_face::ace
      • 'J' means card_face::jack
      • 'Q' means card_face::queen
      • 'K' means card_face::king
      • '2' though '9' are card_face::two through card_face::nine (i.e., the numbers written out in word form)
      • '1' is special since it is the first character of "10" and '1' does not appear on its own --it is followed by a '0' since it represents the number 10 (i.e., card_face::ten).
        • Call the input stream's .peek() member function. This variable will be referred to as 'ch2' below.
        • Check if ch2 is the end-of-file value or not. If it is not, then:
          • Check if ch2 is '0'. If it is then "10" is in the input (i.e., ch and ch2), then the input if "10" (i.e., valid input) so (i) set retval to card_face::ten and (ii) call the input stream's ignore() member function to consume the '0'. If it is not then break out of the switch case.
            • ASIDE: If you want to declare a variable in a switch case statement, you MUST use a block.
      • and the default case should do the following:
        • Call the input stream's unget() function.
        • Call the input stream's setstate() function to fail the stream (i.e., failbit).
  • Outside of the siwtch and if statement, check if retval has a value in it. If not, calll the input stream's setstate() function to fail the stream.
  • Remember to return retval. :-)

Notice the following about the logic of this function:

  • No card_face is returned unless one is properly read in, i.e., invalid input will not produce a card_face.
  • If a '0' does not follow a '1', then the '0' is never consumed (it was only examined with peek()) and '1' is unget() and returned to the stream which means the next read operation will read '1'. (ASIDE: If there was an error on the stream with the .get() call, such would need to be cleared for a subsequent read operation to succeed. This is not done here to for simplicity in this assignment. That said, to be truly robust this should also be properly handled by clearing the stream of errors using .clear(). See std::ios_base which is the parent class of std::istream, e.g., https://en.cppreference.com/w/cpp/io/basic_ios/clear)
  • If the .get() character read in is not a correct character it is always returned to the stream.

This makes the read_card_face() very robust, nicely not consuming stream input when it is invalid. (The latter is not always possible but when it is, such can simplify code that calls such.)

ASIDE 1: For those that took the COMP2140 course. Unformatted functions are easily used with state machine logics for non-deterministic/deterministic finite state automata (NFSAs, DFSAs), regular expressions, grammars (SLR, LR variations, LALR variations, etc.), etc.

class playing_card

Write this class as follows:

  • Declare this as a class --not a struct.
  • This class does not inherit from any other class.
  • Define two private member variables of type:
    • card_face (to store a playing card's face value), and,
    • std::optional<card_suit> (to store a playing card's suit if such exists).
  • Define a public DELETED default constructor.
    • i.e., this will prohibit the default construction of a playing_card.
  • Define a public constructor as follows:
    • playing_card(card_face const& f, std::optional<card_suit> const& s = std::nullopt)
      • i.e., the second argument is defaulted to be an empty std::optional.
    • In the constructor member initialization list, construct the card_face and std::optional<card_suit> members from the constructor arguments' values.
    • Within the constructor body:
      • if std::optional<card_suit> has no value AND the card_face is neither joker value, then throw a std::domain_error() exception passing the string ""playing card without suit must be a joker" to its constructor.
  • Define a public const function called face() that returns the card_face member variable value.
  • Define a public const function called has_suit() that returns, as a bool, whether or not std::optional<card_suit> has a value.
  • Define a public const function called suit() that returns the std::optional<card_suit>'s value.
    • You must not use an if statement to check if std::optional has a value --simply return the value.
    • i.e., this function's precondition is has_suit() == true
  • Publicly define two "= default"ed friend functions with these prototypes:
    • bool operator==(playing_card const&, playing_card const&)
    • std::strong_ordering operator<=>(playing_card const&, playing_card const&)

Notice that no playing_card instances will exist with invalid values since the constructor throws when the value would be invalid. This imples that one can assume any playing_card instance is valid. (This is why there is no "valid()" function for a playing_card.)

NOTE: It is important to ensure member functions are const. For example, the operator<<() output functions accept a const& value to output. If (suitable) functions are not const, then such code could not invoke those functions.

std::optional<playing_card> read_playing_card(std::istream& is)

Like read_card_suit() and read_card_face(), this function will return a std::optional which will have its playing_card value set only if a valid playing card is read in. Write this function as follows:

  • Declare std::optional<playing_card> retval;
  • If read_card_face() returns a card_face value:
    • And if that value is not a joker then:
      • read_card_suit() and if its return value is a card_suit then construct a playing_card passing in the face and the suit and assign such to retval.
    • Otherwise it is a joker,  then assign retval to playing_card only passing in the face value.
  • If retval doesn't have a value set, then call the input stream's setstate() function to fail the stream.
  • Remember to return retval.

std::istream& operator>>(std::istream& is, playing_card& c)

This function is very similar to the one you wrote for card_face earler:

  • Call read_playing_card() with the input stream.
  • If read_playing_card() return has a valid playing_card in it, then assign such to c.
  • Remember to returns the input stream.

std::ostream& operator<<(std::ostream& os, playing_card const& c)

This function is short:

  • Output using operator<<() c.face().
  • If c.has_suit() then output using operator<<() c.suit();
  • Remember to return the output stream.
